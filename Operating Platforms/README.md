#### Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?
The client, The Gaming Room, requested a web-based version of their game Draw It or Lose It to allow multiple teams and players to compete simultaneously across desktop and mobile platforms. They required that each game and team have unique names, that only one instance of the game service exist in memory, and that the system support real-time gameplay, scalability, and future expansions such as additional game modes or chat functionality.
#### What did you do particularly well in developing this documentation?
I excelled in clearly defining the system architecture, domain model, and design patterns. My use of the singleton and iterator patterns, modular class structure, and encapsulation made the design easy to implement while ensuring maintainability and scalability. The platform selection, distributed systems strategy, and security model were particularly well received in feedback.
#### What about the process of working through a design document did you find helpful when developing the code?
Working through the design document was extremely helpful because it forced me to think through the structure and logic of the application before writing any code. By defining the classes, their relationships, and responsibilities ahead of time, I had a clear blueprint to follow, which made coding more efficient and reduced mistakes. The documentation also helped me plan for scalability, unique entity management, and cross-platform functionality, so I could implement features like the Singleton and Iterator patterns correctly the first time. Overall, it provided a roadmap that made coding faster, more organized, and aligned with the client’s requirements.
#### If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?
While the project received excellent feedback, I recognize that the Storage and Memory Management sections could be improved. They currently focus on application-level logic rather than OS-level systems. In the future, I would expand these sections to include considerations like caching strategies, memory allocation, and OS-level file handling to make the documentation more complete.
#### How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?
I prioritized real-time multiplayer support, unique entity management, and cross-platform accessibility in the design. Features such as responsive design, modular class architecture, and safe memory management directly reflect the client’s needs. Considering user needs is crucial because it ensures that the software is not only functional but also reliable, efficient, and user-friendly.
#### How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?
I approached designing the software by first understanding the client’s requirements and then creating a clear domain model using UML diagrams. This helped me visualize the relationships between entities, inheritance, and composition. I also applied object-oriented principles like encapsulation and modularity to keep the design maintainable and scalable. For patterns, I specifically used Singleton for GameService and the Iterator pattern to ensure unique names. In the future, I would continue using UML diagrams for structure and also integrate additional strategies such as flowcharts for data flow, pseudo-code for key algorithms, and design reviews to ensure that all user requirements are accurately interpreted and implemented. These techniques help ensure a smooth transition from design to implementation while minimizing errors.
